1. Application Roles

  1. Employee:
    • Employees can log into the system to manage their leave requests.
    • Leave Request: Employees can apply for different types of leave (e.g., Sick, Paid Time Off (PTO), etc.). They need to specify the leave type, start date (FromDate), end date (ToDate), and a reason or comments for the leave.
    • Leave Status Page: Employees can view the status of their applied leaves (approved, pending, or rejected). They can also see the total leaves they have taken in the current year (financial year: 01-Apr-2024 to 31-Mar-2025).
  2. Manager:
    • Managers have additional responsibilities in the system, as they oversee the leave requests submitted by their team members.
    • Leave Approval/Reject: Managers can view a list of pending leave requests from employees under their supervision. They have the authority to approve or reject these leave requests.
    • Team Leave Overview: Managers can view a list of employees in their team, showing the number of leaves each employee has taken and indicating whether any team member is currently on leave.

2. Database Schema

  • EMPLOYEE Table: Contains information about all employees, including their manager, as every manager is also an employee in the system.
  • LEAVE_REQ Table: Stores details of all leave requests, including the employee ID, leave type, start and end dates, reason, and status (approved, rejected, or pending).
  • LEAVABALANCE Table: Tracks the leave balance for each employee, helping in determining how many leaves an employee has taken and how many are remaining.
  • LOGIN Table: Manages the login credentials for employees, ensuring secure access to the system.

3. Key Features

  1. User Authentication:
    • The application includes a login system where employees and managers authenticate using their credentials.
    • The login is handled through servlets (LoginServlet and LogoutServlet), and sessions are managed for secure access.
  2. Leave Application:
    • Employees can apply for leave, and the system automatically forwards the request to the appropriate manager for approval.
  3. Leave Approval/Management:
    • Managers can see all leave requests from their team members, with options to approve or reject the requests.
  4. Real-time Leave Status:
    • Employees can view the status of their applied leaves in real-time, and managers can get an overview of the leave situation in their teams.
  5. Leave Balances:
    • The system tracks how many leaves an employee has taken and how many are left, based on the organization's leave policy.

4. UI and Frontend

  • Bootstrap UI: The user interface is being built using Bootstrap for a responsive and user-friendly design. The layout includes:
    • Profile Button: Access to employee profile information.
    • Leave Request Section: A form for employees to submit new leave requests.
    • Manager-specific Section: A section visible only to managers where they can approve or reject leave requests.
    • Logout Button: Allows users to securely log out of the application.

5. Development and Testing

  1. API Development:
    • The backend APIs are being developed to handle the various operations, such as applying for leave, approving/rejecting leave, and managing leave balances.
  2. Postman Testing:
    • The APIs are tested using Postman to ensure they function correctly before integration with the UI.
  3. Service Layer:
    • The application follows a service-oriented architecture, with various service layers handling business logic and data processing.
